# Драфт пяти новых секций презентации для SimpleWine

> Этап 1 — контент на ревью. В `index.html` пока ничего не трогаю. После апрува — переношу в HTML с сохранением визуального стиля (Playfair + Inter, бургундская палитра, текущая сетка).

**Принципы по умолчанию (из ответов на старт-вопросы):**
- Цены: ориентир пилота **170к/мес**, минимум **100к/мес**. Цифры в слайдах подобраны мной.
- Юрлицо: упоминаем как «работаем в формате, удобном вашей бухгалтерии — ИП или ООО». Конкретику — в follow-up к Алине.
- Безопасность: показываем то, что **закрыто**. Технические gaps (plain-text ключ, отсутствие SSO, audit-log в roadmap) **не выкладываем** в слайды — переформулируем в «что входит в пилот».
- Self-hosted (Yandex Cloud + Qwen/DeepSeek) — упоминаем как реальную опцию для on-prem-чувствительных шагов.
- Рекомендованный пилот — **Вариант B (SEO + Multiformat)**, обоснование внутри.

## Логика порядка секций в финальной презентации

Текущая структура `index.html`: (1) Title → (2) Больше задач → (3) Как работает → (4) Следующие шаги.

Новые секции встраиваются **между слайдом 3 («Как работает») и слайдом 4 («Следующие шаги»)**, и слайд 4 в финале превращается в «Рекомендация и ближайшие шаги». Порядок:

1. Title (как сейчас)
2. Больше задач (как сейчас)
3. Как работает (как сейчас)
4. **НОВАЯ — Позиционирование** (2 слайда: «платформа, не чатбот» + «межотдельный воркфлоу на примере SEO-статьи»)
5. **НОВАЯ — Сравнение с альтернативами**
6. **НОВАЯ — Безопасность и контур данных** (для IT-директора)
7. **НОВАЯ — Разбивка цены** (отвечает на «почему столько»)
8. **НОВАЯ — Три варианта пилота**
9. Следующие шаги (обновлённый — конкретное предложение по варианту B)

Логика последовательности: сперва говорим, **кто мы** (поз.) → потом **на фоне кого** (сравнение) → закрываем **самый болезненный консерн** (безопасность) → объясняем **цену** → даём **выбор** → конкретный **next step**.

# СЕКЦИЯ 1. Позиционирование

**Закрывает консерны:**
- «По сути, это же обычная нейронка, обычный помощник, кастомизирован под тебя» — Алина, прямая цитата
- «Винлап уже партнёрится с Яндексом, делают автоматизацию контента» — категориальная угроза
- Скрытый страх IT-директора «зачем вообще покупать сторонний продукт, если есть ChatGPT»
- Подспудный вопрос «зачем платить за услугу, если выгоднее посадить копирайтера на ChatGPT»

**Главный месседж секции:** мы — **платформа**, на которую подключаются агенты и строятся воркфлоу между отделами. Это категория, в которой подписка-чат не конкурент.

**Доказательная база (что в коде / гитхабе подтверждает «платформенность»):**

Репозиторий `github.com/cowboybebopbebopbbop/sw` (публичный, упомянут в `audit/00_CONTEXT.md`):

| Артефакт | Где | Что доказывает |
|---|---|---|
| `src/generation/index.js` | оркестратор | Единая точка входа — разные пайплайны через один движок |
| `generateEmail.js` + `generateMultiformat.js` | 2 продуктовых пайплайна | Один engine — несколько разных выходов |
| `promptComposer.js` + `promptComposerMultiformat.js` | composer на формат | Шаги декларируются, не зашиты в монолит |

- ❌ multi-model роутер активно в проде (заявлен архитектурно — Anthropic используется, остальное на полке)

### Слайд 1.1 — «Платформа, не чатбот»

**Заголовок:** «Платформа для контент-производства, не очередной чатбот»

**Лид:**
SimpleWine Studio — это **движок**, на котором уже работают пять разных продуктовых пайплайнов. Каждый новый — это подключённый агент и валидатор, а не «ещё один промпт в окне».

**Контент — двухколоночный `.pos-grid`:**

| **Чатбот / подписка** | **SimpleWine Studio как платформа** |
|---|---|
| Один шаг: запрос → ответ | Оркестратор + подключаемые агенты (`src/generation/index.js`) |
| Каждый формат — заново копи-пейст | Каждый формат — отдельный composer + валидатор как модуль |
| Связь между шагами держит человек | 4-tier knowledge: бренд-правила, лексикон, форматы — данные в git, движок их подхватывает |
| Качество = качество промпта в моменте | Качество = детерминированный валидатор + repair-loop + golden tests |
| Масштабирование = больше людей перед окном чата | Масштабирование = добавить агента в граф, а не нанять копирайтера |

**Под таблицей — strip «Что уже стоит на платформе» (`.platform-chips`):**

`📝 SEO-статьи` · `📦 Multiformat — 65 текстов / прогон` · `✉️ Email-кампании` · `🎨 Изображения через Gemini` · `🧱 HTML-вёрстка статей` · `+ ваш следующий пайплайн` (dashed-чип)

**Footnote:**
Каждый из этих пяти выходов — это не отдельный продукт. Это пайплайн на одном движке. Шестой подключается за дни инженерной работы, а не за месяцы переписывания. Кодовая база: `github.com/cowboybebopbebopbbop/sw` — готовы показать IT-директору.

**Визуал:** `.pos-grid` (уже есть в CSS) + новый `.platform-chips` (бургундная отбивка слева, чипы pill-формы с точкой). Без `.pos-claim`, без `.pos-fit` — для слайда 1.2 нужно место.

**Цитаты, которые закрывает слайд:**
> «По сути, это же обычная нейронка, обычный помощник, кастомизирован специально под тебя на основе твоих кодов»

→ Ответ слайда: нейронка — это компонент. Платформа — это композиция компонентов под доменный процесс.

### Слайд 1.2 — «Где платформа реально шайнит: межотдельный воркфлоу»

**Заголовок:** «Один бриф — четыре роли — готовый продукт»

**Лид:**
Подписка экономит время одного копирайтера. Платформа экономит **цикл согласования между ролями** — а это там, где теряются дни.

**Главный визуал — таблица сравнения процесса (`.process-table`)** на примере SEO-статьи (реальный процесс SW):

| Шаг производства SEO-статьи | Сейчас (4 роли, ручные передачи) | На платформе SimpleWine Studio (бургундовая колонка) |
|---|---|---|
| 01 · Брифы с темами + ключи | Агентство присылает пакет → ждём свободного копирайтера | Бриф загружается в оркестратор пакетом, парсер раскладывает темы |
| 02 · Текст статьи | Копирайтер пишет ~7 ч; редактор может дать правки, может не дать | Текст-агент с лексиконом + ToV из git, детерминированный валидатор compliance |
| 03 · Иллюстрации | Дизайнер: постановка ТЗ → очередь → отрисовка → согласование | Image-агент через Gemini, привязка к тексту автоматически |
| 04 · Вёрстка | Верстальщик собирает HTML → проверка отображения → правки | Layout-агент → готовый HTML в том же артефакте |
| 05 · Готово к публикации | Финальная сборка пакета вручную | Один артефакт-пакет на выходе оркестратора |

**Footer таблицы (бургундовая отбивка):**

| Итого ролей в цикле | 4 роли · 4 точки передачи | **1 пайплайн · 0 ручных передач** |

**Под таблицей — claim-полоса (бургундовый блок):**

> *ROI подписки* = время одного человека.  *ROI платформы* = весь цикл согласования между ролями.

**Footnote (честность — что НЕ автомат):**
- Тему и стратегию задаёт человек (контент-маркетолог)
- Финальное «опубликовать» — человек, не пайплайн
- Платформа не заменяет роли — она убирает простои между ними
- Тот же подход применим к multiformat-пакетам, email-кампаниям и любым новым процессам, где сейчас цепочка ролей передаёт работу из рук в руки

**Визуал:** новый `.process-table` (compact, бургундовая правая колонка как акцент) + новый `.platform-claim` (бургундовая полоса с italic Playfair). Использует тот же визуальный язык, что и `.audit-table` в секциях «Сравнение» / «Цена».

**Цитата meeting.txt, которую закрывает слайд:**
> «Мне нужно не просто чтобы статья быстрее писалась — мне нужно, чтобы её сразу можно было запустить»

→ Ответ слайда: платформа закрывает не «текст», а весь путь до готового к публикации артефакта.

### Слайд 2.2 — Окно vs runtime (нарративная развязка сравнения)

**Заголовок:** «Подписка — это окно. Пилот — это runtime»

**Ключевой тезис (одна метафора, развёрнутая в 4 строки):**

> ChatGPT, Custom GPT, AlpinaGPT, YaGPT — это **окна для разговора с моделью**. Кто-то открывает окно, копирует бриф, копирует ответ, проверяет вручную, вставляет в Mindbox.
> SimpleWine Studio — это **runtime**: бриф входит в систему, готовый артефакт выходит. Compliance, согласованность форматов, бренд-лексикон — внутри пайплайна, не в голове копирайтера.

**Два визуальных flow (стилистика как существующий `.flow` блок на слайде 3):**

**С чат-инструментом:**
`Бриф → копи-пейст в чат → ручная проверка лимитов → ручная проверка алкогольного компла́йенса → ручная сверка скидки между форматами → копи-пейст в Mindbox` — **6 ручных шагов, точка отказа на каждом**

**С SimpleWine Studio:**
`Бриф → парсер → генератор → валидатор → готовый артефакт` — **0 ручных шагов до выхода готового пакета**

**Визуал:** два горизонтальных flow один под другим, верхний — серый/нейтральный с красными точками отказа, нижний — бургундовый.

**Footnote:**
> Vinlab × Yandex анонсировали партнёрство, но публичных продуктовых результатов пока нет. Окно для решения — сейчас, не через квартал.

### Слайд 3.1 — Карта данных: что куда уходит

**Заголовок:** «Что покидает контур SimpleWine, а что — никогда»

**Ключевой тезис:** Прозрачная карта потока данных. Никакого маркетингового пуха, только факт.

**Контент — диаграмма + таблица:**

**Что НЕ покидает контур SimpleWine (бургундовая колонка):**
- Базы клиентов (CRM, Mindbox-сегменты)
- Персональные данные клиентов (ФИО, телефон, email)
- Заказы, чеки, история покупок
- Внутренние финансовые показатели
- Метрики Яндекс.Метрики и аналитика
- Фотографии, исходники макетов

**Что покидает контур SimpleWine (нейтральная колонка):**
- **Только текст брифа** (продукт, акция, % скидки, целевая аудитория в обобщённой формулировке)
- Никаких ПДн в брифе **не должно быть** — это контролируется на этапе ввода

**Куда уходит и через что:**

```
Браузер копирайтера SW
        ↓ HTTPS / TLS 1.3
Шлюз API (multi-model роутер)
        ↓ HTTPS / TLS 1.3
LLM-провайдер (Anthropic Claude / Yandex GPT — по выбору)
        ↓
Готовый текст возвращается в браузер
```

**Визуал:** flow-диаграмма в стиле существующего `.flow`, но вертикальная и с двумя цветовыми кольцами: «внутренний контур SW» (бургундовый) и «внешний контур (LLM)» (нейтральный).

**Цитата из meeting.txt:**
> «Какие-то персональные данные, метрики, они не покидают то место, в котором они находятся»

### Слайд 3.2 — Чек-лист для IT-директора

**Заголовок:** «10 вопросов от ИБ — 10 ответов**

**Формат:** двухколоночная таблица (вопрос — ответ), сверстана как `.compare` без бургундовой заливки.

| Вопрос ИБ | Что обеспечено в пилоте |
|---|---|
| Где физически хранятся брифы? | Не хранятся в нашей системе. Передаются транзитом, после генерации не персистятся на нашей стороне |
| Шифрование канала? | TLS 1.3, end-to-end по всему маршруту |
| Кто имеет доступ к брифам? | Только аутентифицированный пользователь SW в своей сессии. Изоляция тенанта на уровне ключа |
| Учит ли LLM-провайдер модель на наших данных? | Anthropic — нет (no-training by default, фиксируется в договоре). Yandex GPT — нет, по корп-договору |
| Аудит-лог обращений | Append-only лог в Yandex Object Storage, экспорт в PDF — **входит в пилот** |
| Аутентификация | OAuth 2.0 + опционально SSO через Keycloak / корп. AD — **входит в пилот** |

### Слайд 5.1 — Три варианта пилота на выбор

**Ключевой тезис:** Не «три тарифа», а три **последовательные ступени**. На любой можно остановиться.

**Контент — три карточки бок о бок (как `.steps`, но три колонки):**

#### Вариант A — SEO Sprint
- **Цель:** проверить, реально ли получить X2-X4 рост объёма SEO без потери качества
- **Объём:** 100 SEO-статей за 1 месяц, 1 копирайтер
- **Сроки:** 1 месяц

- **Цель:** проверить, что система решает не только SEO, но и закрывает рутину по push/sms/баннерам — то есть «всю» текстовую работу контент-отдела

- **Что входит:** всё из A + multiformat (65 текстов одним прогоном с гарантией compliance) + расширение валидатора под новые форматы по запросу + аудит-лог + базовый pre-processing брифа (защита от ПДн)
- **Чего НЕТ:** интеграции с Mindbox/Bitrix (отложены в C), модерация отзывов
- **Цена:** **~170 000 ₽/мес**
- **Критерии успеха:**
  - 200 статей за месяц при ≤ 2.5 ч/статью

  - Замер высвобожденного времени на разработку новых форматов / тестов
- **Что получает SW на выходе:** работающий конвейер для всего поточного контента + RUNBOOK + golden tests + понимание реальной стоимости масштабирования на 3+ копирайтеров

- **Что получает SW на выходе:** полнофункциональный production runtime контента + договор на пост-пилотную поддержку
